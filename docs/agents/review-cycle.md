# Review Cycle Workflow

This workflow applies when an issue has the `implemented: ready for review`
progress label and needs to move toward `implemented: complete`.

## Goal

Move an implemented issue through a repeatable review loop before closing it.

`implemented: ready for review` means work exists and is ready for maintainer
or reviewer evaluation.

`implemented: complete` means the implementation has been accepted by the
maintainer or merged.

Do not close an issue until `implemented: complete` is present.

## Entry Criteria

Start this workflow only when the issue has:

- Exactly one category label: `bug` or `enhancement`
- Exactly one state label: `ready-for-agent` or `ready-for-human`
- The progress label `implemented: ready for review`

If labels conflict, stop and ask the maintainer.

## Gather Context

Read the full review context before deciding an outcome:

- Issue body, labels, comments, and agent brief
- Linked PR, branch, commits, or implementation notes
- Acceptance criteria and explicit maintainer decisions
- Relevant PRD, plan, `CONTEXT.md`, and ADRs
- Test output, CI status, screenshots, or manual QA notes

If the issue is `ready-for-human`, preserve that state unless the maintainer
explicitly reclassifies it. Human judgment, credentials, product approval,
manual QA, legal/fiscal review, or vendor decisions must be accepted or waived
by the maintainer before completion.

## Review Loop

1. Verify that the implementation matches the issue acceptance criteria.
2. Inspect the branch, PR, or diff for behavior, quality, security, migrations,
   documentation, and localization concerns.
3. Run relevant checks when feasible. If checks cannot be run, document why.
4. If problems exist, post actionable review notes and keep
   `implemented: ready for review`.
5. If changes are made after review, repeat the review loop from the top.
6. When the maintainer accepts the implementation or the PR is merged, add
   `implemented: complete`.
7. Close the issue only after `implemented: complete` is present.

## Required Checks

Before marking complete, confirm:

- Acceptance criteria are satisfied or explicitly waived by the maintainer.
- Relevant tests/checks passed, or skipped checks are documented with a reason.
- No sensitive data, `.env*` files, credentials, or provider secrets were
  committed.
- User-facing strings use the project's localization or translation system when
  one exists.
- Database migrations, if present, are documented and match schema changes.
- Shared contracts or types are updated before duplicated downstream shapes
  appear in apps.
- Any PR or branch notes explain what changed and how it was verified.

## Review Outcomes

### Changes Requested

Use this when the implementation is close but not complete.

Keep:

- Original category label
- Original state label
- `implemented: ready for review`

Do not add `implemented: complete`.

Post a comment that includes:

- What failed review
- Why it matters
- Specific files, behavior, or acceptance criteria involved
- Concrete next steps

Use [`review-feedback-workflow.md`](./review-feedback-workflow.md) to address
the feedback and automatically trigger re-review after fixes.

### Accepted

Use this only when the maintainer accepted the work or the PR was merged.

Apply:

- `implemented: complete`

Then close the issue with a short summary of:

- What was completed
- How it was verified
- PR, branch, or commit reference

## GitHub Commands

List issues ready for review:

```bash
gh issue list --state open --label "implemented: ready for review"
```

Read a specific issue and its comments:

```bash
gh issue view <number> --comments
```

Add the completion label:

```bash
gh issue edit <number> --add-label "implemented: complete"
```

Close after completion:

```bash
gh issue close <number> --comment "<summary>"
```

## Issue Comment Template

Every review comment posted to the issue tracker must start with the standard
AI disclaimer:

```
> *This was generated by AI during review.*

## Review Result

**Outcome:** Changes requested | Accepted

**What was reviewed:**

- Acceptance criteria
- Tests/checks
- Code quality
- Documentation and migration notes

**Findings:**

- Finding 1
- Finding 2

**Verification:**

- Command/result 1
- Command/result 2

**Next step:**

- Required follow-up or closure note
```
