# Review Feedback Workflow

This workflow applies when an issue or linked PR has review feedback that must
be addressed before the issue can move from `implemented: ready for review` to
`implemented: complete`.

Use this workflow for issue comments, PR review comments, maintainer notes, and
reviewer subagent findings that request changes.

## Goal

Turn review feedback into a concrete fix checklist, apply the fixes, verify
them, and automatically trigger a fresh review pass.

Do not ask whether to re-run review after fixing review feedback. Re-review is
required unless the maintainer explicitly pauses the cycle.

## Entry Criteria

Start this workflow when:

- The issue has `implemented: ready for review`.
- The latest review outcome is `Changes requested`, or an issue/PR comment asks
  for concrete follow-up.
- The feedback is actionable enough to fix without a new product decision.

Stop and ask the maintainer when:

- Feedback conflicts with the issue acceptance criteria.
- Feedback requires product, legal, fiscal, vendor, credential, or manual QA
  judgment.
- State or category labels conflict.
- The branch or PR that contains the implementation cannot be identified.

## Gather Feedback

Read the full issue and linked PR context:

```bash
gh issue view <number> --comments
```

If there is a linked PR, inspect its review threads, comments, checks, and diff:

```bash
gh pr view <number> --comments --json number,title,state,headRefName,reviewDecision,comments,reviews,statusCheckRollup
gh pr diff <number>
```

Build a checklist from the newest unresolved feedback. Include:

- The source of each finding: issue comment, PR review, CI failure, or subagent
  review.
- The exact behavior, file, test, or acceptance criterion involved.
- The expected fix or decision needed.
- Whether the finding is blocking completion.

Do not re-open feedback that was already resolved by a newer comment, commit,
or maintainer decision.

## Fix Loop

1. Confirm you are on the implementation branch or the branch requested by the
   maintainer.
2. Preserve unrelated local changes.
3. Apply one focused set of fixes at a time.
4. Use the `tdd` skill for runtime behavior changes.
5. Update docs, translations, contracts, migrations, or tests when the feedback
   requires it.
6. Run the smallest relevant verification command first.
7. Broaden verification before requesting re-review.

If the feedback spans independent areas, use the `subagent-driven-development`
skill where it applies. Follow the same policies as
[`implementation-workflow.md`](./implementation-workflow.md):

- Run spec compliance review before code quality review.
- Use fresh subagents for focused review or fix tasks.
- Do not run multiple implementer subagents in the same worktree at the same
  time.
- Parallel exploration or review is allowed when write scopes do not conflict.
- Fix and re-review until reviewers pass or a blocker requires maintainer input.

## Verification Before Re-Review

Before triggering review again, record:

- Feedback items fixed.
- Files or behavior changed.
- Tests/checks run and their result.
- Checks that could not run and why.
- Any remaining risk or maintainer decision needed.

Use repo-standard commands where relevant. Examples:

```bash
<project lint command>
<project typecheck command>
<project test command>
```

For narrow changes, prefer scoped checks first, then broaden when the change
touches shared behavior, contracts, migrations, or user-facing workflows.

## Automatic Re-Review Trigger

After fixes and verification, trigger review again without waiting for another
maintainer prompt.

Use this order:

1. Re-run the same reviewer type that found the issue.
2. If that reviewer passes, run the next required reviewer in the review chain.
3. For implementation work, run spec compliance review before code quality
   review.
4. Use `subagent-driven-development` for independent review/fix tasks when it
   can reduce review latency without
   overlapping writes.
5. Repeat fix and re-review until all reviewers pass or a blocker requires
   maintainer input.

If review is happening through issue comments rather than an active subagent,
post a re-review handoff comment on the issue. The comment must start with the
standard AI disclaimer:

```
> *This was generated by AI during review feedback.*

## Review Feedback Addressed

**Fixed:**

- Finding 1
- Finding 2

**Verification:**

- Command/result 1
- Command/result 2

**Ready for re-review:**

- Spec compliance reviewer
- Code quality reviewer

**Notes:**

- Remaining risk, skipped check, or maintainer decision if any
```

Only mark the issue `implemented: complete` after the maintainer accepts the
work or the PR is merged. Passing automated or subagent re-review alone is not
completion.

## Reviewer Prompts

When no more specific project prompt exists, use these reviewer roles.

### Spec Compliance Reviewer

Ask the reviewer to verify:

- The implementation satisfies the issue acceptance criteria.
- The implementation does not add out-of-scope behavior.
- Domain terms match `CONTEXT.md`.
- User-facing behavior matches linked PRD, plan, and maintainer decisions.
- Missing acceptance criteria, skipped checks, or ambiguity are explicitly
  called out.

The reviewer must return one of:

- `PASS`
- `NEEDS_CHANGES`
- `BLOCKED`

### Code Quality Reviewer

Ask the reviewer to verify:

- Type safety, error handling, security, and data exposure.
- Tests cover the changed public behavior.
- API contracts and shared types are updated before downstream duplication.
- Migrations match schema changes and are documented.
- UI strings use the translation system.
- The diff avoids unrelated refactors and preserves existing patterns.

The reviewer must return one of:

- `PASS`
- `NEEDS_CHANGES`
- `BLOCKED`

## Outcomes

### Re-Review Passed

Keep `implemented: ready for review` until maintainer acceptance or PR merge.

Post or update review notes with:

- Reviewers run
- Verification commands
- Remaining manual checks, if any
- Link to the PR, branch, or commit

### More Changes Requested

Keep `implemented: ready for review`.

Add the new findings to the checklist and repeat the fix loop.

### Blocked

Do not guess.

Post the blocker with:

- What is blocked
- Why it cannot be resolved by the agent
- The exact maintainer decision or external access needed

## Completion Boundary

This workflow prepares an issue for acceptance. It does not complete the issue
by itself.

Use [`review-cycle.md`](./review-cycle.md) to move from passed review to
`implemented: complete` and close the issue after maintainer acceptance or PR
merge.
